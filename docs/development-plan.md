# 开发路线

## 阶段 0：仓库与设计

- 初始化 Git 仓库。
- 固定 MVP 范围。
- 固定技术栈。
- 写同步协议和系统架构。
- 建立项目骨架。

验收标准：

- 仓库有初始提交。
- 关键功能都有设计入口。
- 能按文档拆分后续任务。

## 阶段 1：播放器单机能力

- 创建 Tauri 桌面应用。
- 接入 mpv/libmpv。
- 实现本地文件打开、播放、暂停、seek、倍速。
- 实现字幕加载和字幕偏移。
- 实现快捷键。
- 实现本地播放列表。

验收标准：

- 单机可稳定播放常见 `mp4/mkv`。
- 控制栏和快捷键都能驱动播放器。
- 字幕偏移可立即生效。

## 阶段 2：信令服务与房间

- 创建 Axum WebSocket 服务。
- 实现房间创建、加入、离开。
- 实现 sessionId 和短期重连。
- 实现 presence 和角色状态。

验收标准：

- 两个客户端能进入同一房间。
- 断线 30-120 秒内重连恢复成员身份。

## 阶段 3：文件匹配

- 读取媒体元数据。
- 实现快速匹配。
- 实现分段 hash。
- 实现严格 hash。
- 实现 `mediaId -> localPath` 本地绑定。

验收标准：

- 两端相同文件能自动匹配。
- 文件名不同但内容相同可匹配。
- 文件不一致时明确提示。

## 阶段 4：同步播放

- 建立 WebRTC DataChannel。
- 定义可靠有序控制通道。
- 实现播放命令广播与应用。
- 实现房间时间同步。
- 实现漂移检测、微调和 seek 校准。

验收标准：

- 播放、暂停、seek 在两端同步。
- 正常网络下漂移长期小于 250ms。
- 断线重连后能恢复到正确进度。

## 阶段 5：模式、列表和下一集

- 实现主控模式。
- 实现自由模式和冲突处理。
- 实现房间播放列表同步。
- 实现剧集识别和自动下一集。
- 实现手动下一集。

验收标准：

- 主控模式下非主控不能覆盖播放状态。
- 自由模式下冲突有稳定结果。
- 下一集切换前能检查双方是否都有文件。

## 阶段 6：互动与打磨

- 实现同步表情反应。
- 连接状态、重连状态、延迟状态可视化。
- 完善错误提示。
- 增加基础遥测日志，本地优先。

验收标准：

- 表情反应不影响播放同步。
- 常见异常有明确 UI 状态。

## 阶段 7：发布准备

- 打包 Windows 版本。
- 增加自动更新设计。
- 增加隐私说明。
- 增加端到端测试脚本。

验收标准：

- 可以交给真实用户安装试用。
- 初始隐私和安全边界明确。
